## Для чего нужен Datagon

Система помогает команде отвечать на вопросы:

- какие **цены у конкурентов** по нашим SKU и артикулам;
- что **в наличии** у нас и у конкурентов (по данным парсинга);
- какие пары товаров **уже сопоставлены** автоматически и что нужно **проверить вручную**;
- готовы ли данные к работе с **МойСклад** (`ms_export`).

## Что умеет система (по блокам)

| Блок | Смысл |
|------|--------|
| **Конкуренты** | Учёт сайтов конкурентов и **CSS-селекторов** для разбора карточек товара. |
| **Очередь** | Список **URL** на парсинг, статусы, массовый запуск и сброс. |
| **Результаты** | Таблица **`prices`**: что парсер сохранил (цена, SKU, время, проект). |
| **Мои сайты** | Подключение **ваших** БД (Bitrix/Webasyst) как источников каталога. |
| **Мои товары** | Локальный каталог **`my_products`**, фильтры, сравнение с конкурентами и МС. |
| **Сопоставление** | Автопоиск пар «наш ↔ конкурент», подтверждение/отклонение. |
| **МойСклад** | Синхронизация номенклатуры в **`ms_export`**, фильтры, статус задач. |
| **Логи** | Сводка **фоновых** задач (синк, парсинг, матчинг, МС). |
| **Настройки** | Лимиты парсинга, паузы, пользователи, токены. |

## Как пользоваться этой справкой

1. **Боковое меню «Разделы»** — дерево всех страниц; на мобильном открывается кнопкой «гамбургер» в шапке.
2. **Поиск** в навбаре — локальный индекс по тексту страниц (нужна актуальная сборка, чтобы существовал `search-index.json`).
3. **Содержание справа** (на широком экране) — якоря по заголовкам текущей страницы.
4. Переключатель **светлая/тёмная тема** — в шапке (если не отключён темой ОС).
5. Термины вроде `my_products`, **очередь**, **матчинг** — см. отдельную страницу [Глоссарий](/docs/glossary/).

## Основной сценарий (развёрнуто)

### Шаг 1. Конкуренты

Создайте **проект** для каждого конкурента: домен, **селектор цены** (обязателен), при возможности — название, SKU, признак «нет в наличии». Проверьте селекторы на **одной** реальной карточке в DevTools браузера. Подробно: [Конкуренты](/docs/projects/).

### Шаг 2. Очередь URL

Добавьте ссылки на карточки товаров (вручную, пакетом или через обход sitemap — если доступно). Убедитесь, что URL относятся к **правильному проекту**. Подробно: [Очередь](/docs/queue/).

### Шаг 3. Парсинг и результаты

Запустите парсинг для выбранных или отфильтрованных строк. Смотрите **статус** в очереди; при `error` — текст причины, правка селекторов, **reset** в `pending` и повтор. Итоговые строки — в [Результаты](/docs/results/).

### Шаг 4. Мои сайты

Подключите **источник** (вашу БД), проверьте **маппинг полей**, сохраните и выполните **синхронизацию** в `my_products`. Подробно: [Мои сайты](/docs/mysites/).

### Шаг 5. Сопоставление

Выберите **свой сайт** и **конкурентов**, задайте порог и размер пакетов, запустите матчинг. Разберите **pending**-кандидатов. При «зависании» — [Логи](/docs/processes/) и FAQ ниже. Подробно: [Сопоставление](/docs/matches/).

### Шаг 6. Мои товары

Контролируйте **активность**, **связь с МС**, **разрыв цен** с конкурентами, точечные действия по строкам. Подробно: [Мои товары](/docs/myproducts/).

### Шаг 7. МойСклад (по необходимости)

Запустите синхронизацию с API МС, дождитесь завершения, проверьте `ms_export`. Подробно: [МойСклад](/docs/moysklad/).

**Краткая цепочка экранов:** Конкуренты → Очередь → Результаты → Мои сайты → Сопоставление → Мои товары → МойСклад.

## Сценарии «первый день»

1. **Только конкуренты (без своего каталога):** [Конкуренты](/docs/projects/) → [Очередь](/docs/queue/) → [Результаты](/docs/results/). Настройте задержки в [Настройках](/docs/settings/), следите за [Логами](/docs/processes/), если включён массовый парсинг.
2. **Свой каталог из БД:** [Мои сайты](/docs/mysites/) (подключение и синк) → [Мои товары](/docs/myproducts/). Затем при необходимости [МойСклад](/docs/moysklad/) для сверки с учётной системой.
3. **Цены «мы vs конкурент»:** п.1 + актуальные `my_products` (п.2) → [Сопоставление](/docs/matches/) (ручное подтверждение пар) → снова **Мои товары** для колонок разрыва.
4. **Продакшен после установки:** [Деплой](/docs/deploy/), проверка `curl` из FAQ, смена пароля admin, токен МС в `config.js` / `MS_TOKEN`.

## Технологии и репозиторий

- **Сервер:** Node.js, Express, MySQL, Axios, Cheerio, bcrypt.
- **Панель:** статика в `public/` (ArchitectUI Bootstrap).
- **Справка:** Docusaurus 3; сборка: `npm run docs:docusaurus:build`.

| Путь | Назначение |
|------|------------|
| `server.js`, `routes/`, `lib/` | API и логика |
| `public/*.html`, `public/datagon-vanilla.js` | собранная панель |
| `static-html/vanilla/` | исходники HTML → `npm run sync:vanilla-public` |
| `worker.js`, `sync-worker.js` | фон |
| `config.js` | БД, порт, `MS_TOKEN` / `msToken` |

## Быстрый старт (локально)

1. `git clone` репозитория, `cd` в каталог проекта.
2. `npm install`.
3. Скопируйте/настройте `config.js`: MySQL (`host`, `user`, `password`, `database`), `port`, при необходимости токен МойСклад.
4. `npm start`, откройте `http://localhost:<port>/login.html` или дашборд.
5. Документацию собрать: `npm run docs:docusaurus:build` (опционально).

После правок **серверного** кода перезапустите процесс Node — см. раздел **«Важные замечания»** ниже.

## FAQ и типовые проблемы

### 1) Не удаётся войти

Проверьте:

- URL панели: `…/dashboard.html` или корень `/` с редиректом;
- запущен ли **`npm start`** (или pm2 на сервере);
- логин и пароль; смена пароля через UI или API.

Проверка API:

```bash
curl -i -X POST "http://localhost:3000/api/auth/login" \
  -H "Content-Type: application/json" \
  --data '{"username":"YOUR_USERNAME","password":"YOUR_PASSWORD"}'
```

### 2) Приложение не стартует: ошибка БД

В `config.js`: `db.host`, `db.user`, `db.password`, `db.database`; доступность MySQL и права; существование базы. В консоли ищите **`[DB] Fatal Error`**. Исправьте параметры и **перезапустите** Node.

### 3) Источник не добавляется (DB Connect)

Обычно: внешняя БД недоступна с сервера Datagon, неверный **хост/порт**, **имя базы**, **таблица товаров**, поля SKU/цены/остатка. Для **Webasyst** — отдельная таблица SKU и поля `wa_*`. См. [Мои сайты](/docs/mysites/).

### 4) Парсинг не даёт данных

Частые причины: неверные **селекторы**; страница не карточка товара; цена только в **JS** (в сыром HTML нет); лимиты/бан со стороны сайта.

Действия: проверить одну страницу через API `POST /api/pages/page/:id`; заполнить `selector_price`, `selector_name`, `selector_sku` в проекте; увеличить **`page_delay_ms`** в [Настройках](/docs/settings/).

### 5) В очереди много `error`

Отфильтровать ошибки, прочитать причину, исправить URL/селекторы, выполнить **reset** в `pending`, снова запустить парсинг. См. [Очередь](/docs/queue/).

### 6) Сопоставление не стартует или «зависло»

Если ответ API говорит, что задача уже идёт:

- `GET /api/matches/status?my_site_id=…`
- `POST /api/matches/stop` с телом `{ "mySiteId": … }`
- при необходимости `POST /api/matches/retry-last`

На больших объёмах уменьшите **`batchSize`**, увеличьте **`batchPauseMs`**, сузьте выборку (`productSearch`, список `productIds`). См. [Сопоставление](/docs/matches/).

### 7) После синка часть «моих товаров» неактивна

Это **нормально**: перед синком позиции помечаются `is_active=0`; найденные в источнике становятся `1`; отсутствующие в источнике остаются неактивными.

### 8) Не работает МойСклад

Токен: переменная **`MS_TOKEN`** или `msToken` в `config.js`, права API МС. Контроль: `POST /api/ms/sync`, `GET /api/ms/status`, `GET /api/ms/export`. См. [МойСклад](/docs/moysklad/).

### 9) После деплоя старый фронт

Жёсткое обновление страницы (**Cmd+Shift+R** на macOS), `git pull` на сервере, **`pm2 restart`** (или ваш процесс), убедиться что выложен актуальный `public/`. Чеклист: [Деплой](/docs/deploy/).

### 10) Высокая нагрузка MySQL

Не путать **очередь страниц** (`pages`) и тяжёлые **списки товаров** / **`GET /api/my-products/stats`**. После обновления кода с оптимизациями — **перезапуск Node**, не только MySQL. Диагностика: `SHOW FULL PROCESSLIST`, при необходимости `EXPLAIN`.

## Важные замечания

- Таблицы БД создаются и дорабатываются при старте `server.js`.
- Совместимость: **`POST /api/login`** (legacy) и **`POST /api/auth/login`**.
- После изменений **`server.js`**, **`routes/`**, **`lib/`** нужен **перезапуск Node** — иначе API в браузере останется старым.

Полный дубль части материала с дополнениями по эксплуатации — в **`README.md`** в корне репозитория (разделы «FAQ», «Структура», «Деплой»).
