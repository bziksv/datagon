**`/matches.html`** — настройка и запуск **фонового сопоставления**: для выбранного [Моего сайта](/docs/mysites/) и набора **проектов конкурентов** система ищет пары с оценкой похожести (SKU, название и т.д. по логике сервера). Результат — список кандидатов со статусами **pending** / **confirmed** / **rejected**.

## Подготовка к запуску

1. В [Мои сайты](/docs/mysites/) выполнен **синк** — в `my_products` есть актуальные строки.
2. В [Результаты](/docs/results/) есть **свежие** `prices` по выбранным конкурентам.
3. В [Настройках](/docs/settings/) заданы разумные **лимиты** нагрузки (если матчинг длинный — уменьшите `batchSize`, увеличьте паузы).

## Параметры задачи (что означают)

| Параметр | Смысл |
|----------|--------|
| **Порог** (`threshold`) | Минимальная уверенность для кандидата; выше — меньше шум, ниже — больше кандидатов на ручную проверку. |
| **batchSize** | Сколько пар обрабатывать за один проход цикла; влияет на нагрузку CPU/БД. |
| **batchPauseMs** / micro-паузы | Паузы между пакетами и микро-паузы внутри длинных циклов — защита от перегрева БД. |
| **Режим / выбор товаров** | «Все», по поиску, по списку id — зависит от формы в UI. |

## После запуска

1. Откройте [Логи](/docs/processes/) — убедитесь, что задача **в работе** или **завершена**, а не упала с ошибкой.
2. Обновите список кандидатов в разделе сопоставления.
3. Для каждой **pending**-строки: **Подтвердить** (если пара верная) или **Отклонить** (если ложное срабатывание).

## Если «задача уже выполняется»

- `GET /api/matches/status?my_site_id=…` — посмотреть состояние.
- `POST /api/matches/stop` с `{ "mySiteId": N }` — остановить.
- `POST /api/matches/retry-last` — продолжить/повторить последнюю конфигурацию (см. [API](/docs/api/#matches)).

Не запускайте второй матчинг для того же `my_site_id`, пока первый не завершён — сервер вернёт ошибку.

## Ручной разбор кандидатов

- **Pending** — не «ошибка», а ожидание решения: просмотрите пару SKU/названий и контекст цены в [Результаты](/docs/results/) по конкуренту.
- **Confirmed** — пара участвует в расчётах в [Мои товары](/docs/myproducts/) (колонки конкурента, разрыв цены) согласно логике сервера.
- **Rejected** — ложное срабатывание; при повторном матчинге та же пара может появиться снова, если изменились данные или порог.
- При **сотнях** pending используйте сортировку/фильтр в UI (если есть) или выгрузку через [`GET /api/matches/list`](/docs/api/#matches) с пагинацией.

## Когда снижать порог, а когда повышать

- **Повысить** `threshold`, если много явного мусора в pending — меньше кандидатов, выше точность.
- **Понизить**, если «тихо» и кандидатов мало при заведомо похожих названиях — осторожно, растёт ручная работа.

## API (основное)

- `POST /api/matches/start-matching` — тело с `mySiteId`, `competitorIds`, `threshold`, `batchSize`, …
- `GET /api/matches/list` — список пар с фильтром по статусу.
- `POST /api/matches/confirm` / `reject` — тело `{ "id": … }`.

Полный перечень — [REST API — Matches](/docs/api/#matches).
