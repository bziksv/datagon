# p.datagon.ru — парсер цен и сопоставление товаров

Веб-приложение для сбора цен конкурентов, синхронизации собственного каталога из внешних БД (Bitrix/Webasyst), автоматического сопоставления товаров и подготовки выгрузки для МойСклад.

## Для чего нужен проект

Система помогает команде быстро отвечать на вопросы:

- какие цены у конкурентов по нашим SKU;
- какие товары в наличии у нас и у конкурентов;
- какие позиции уже сопоставлены автоматически, а какие требуют ручной проверки;
- какие данные готовы к дальнейшей обработке в МойСклад.

## Что умеет система

- ведение списка конкурентных сайтов и CSS-селекторов для парсинга;
- очередь URL-страниц с пакетной обработкой и повторным запуском;
- сохранение результатов парсинга цен и наличия;
- подключение "моих сайтов" (источников) через доступ к их БД;
- синхронизация локальной таблицы `my_products`;
- умное сопоставление товаров (по SKU и по названию с оценкой похожести);
- подтверждение/отклонение найденных сопоставлений;
- экспорт данных в `ms_export` через API МойСклад.

## Технологии

- Node.js + Express
- MySQL (`mysql2/promise`)
- Axios + Cheerio (HTTP и парсинг HTML)
- bcryptjs (аутентификация)
- Статический фронтенд в `public/` (HTML/CSS/JS)

## Структура проекта

- `server.js` — запуск приложения, инициализация БД, подключение роутов.
- `routes/` — серверные API-модули.
- `public/` — пользовательский интерфейс (legacy HTML/JS и собранный React в `public/architectui-react-pro/`).
- `architectui-react-pro/` — исходники SPA (CRA), сборка в `public/architectui-react-pro/` через `npm run build:datagon-spa`.
- `worker.js`, `sync-worker.js` — фоновые скрипты пакетной обработки.
- `config.js` — порт, доступ к БД, токен МойСклад.
- `scripts/` — утилиты сборки, скриншотов документации, smoke e2e (см. `scripts/README.md`).
- `docs-docusaurus/` — исходники пользовательской документации (Docusaurus).
- `docs/repo/` — документация для разработчиков: API, деплой, план миграции UI.

## Первый вход

Для входа используйте учетную запись, настроенную в вашей среде.
Рекомендуется регулярно обновлять пароль через API/UI.

## Основной пользовательский сценарий

1. Добавьте проекты конкурентов в разделе проектов.
2. Загрузите URL-страницы в очередь парсинга.
3. Запустите парсинг страниц и проверьте результаты.
4. Добавьте "мои сайты" (источники товаров) и выполните синхронизацию.
5. Запустите сопоставление товаров и разберите найденные пары.
6. При необходимости запустите синхронизацию в МойСклад и проверьте экспорт.

## Разделы API

Подробное описание всех методов и параметров — в [`docs/repo/API.md`](docs/repo/API.md).

Коротко по группам:

- `/api/auth` — вход и смена пароля
- `/api/settings` — рабочие настройки системы
- `/api/projects` — конкуренты
- `/api/pages` и `/api/parse` — очередь парсинга URL
- `/api/results` — результаты парсинга
- `/api/my-sites` — источники собственных товаров
- `/api/my-products` — локальный каталог
- `/api/matches` — сопоставление товаров
- `/api/ms` — экспорт в МойСклад
- `/api/sync-*` — глобальная синхронизация

## FAQ и типовые проблемы

### 1) Не удается войти в систему

Проверьте:

- правильный URL: `http://localhost:3000/index.html`;
- что сервер запущен (`npm start`);
- корректность логина и пароля.

Быстрая проверка API входа:

```bash
curl -i -X POST "http://localhost:3000/api/auth/login" \
  -H "Content-Type: application/json" \
  --data '{"username":"YOUR_USERNAME","password":"YOUR_PASSWORD"}'
```

### 2) Ошибка подключения БД проекта (приложение не стартует)

Проверьте `config.js`:

- `db.host`, `db.user`, `db.password`, `db.database`;
- доступность MySQL и права пользователя;
- что нужная БД существует.

Если в консоли есть `[DB] Fatal Error`, исправьте параметры подключения и перезапустите сервер.

### 3) Источник (`my-sites`) не добавляется: DB Connect Error

Причина обычно в недоступности внешней БД источника или в неверных полях таблицы.

Проверьте:

- хост/порт/логин/пароль внешней БД;
- правильность имени таблицы товаров;
- корректность полей для SKU/цены/остатка;
- для Webasyst: таблицу SKU и поля `wa_*`.

### 4) Парсинг страниц не дает результатов

Частые причины:

- неверные CSS-селекторы в проекте конкурента;
- страница не является карточкой товара;
- цена на сайте рендерится JS и отсутствует в HTML-ответе;
- сайт ограничивает запросы.

Что сделать:

- проверить вручную одну страницу через `POST /api/pages/page/:id`;
- убедиться, что в проекте заполнены `selector_price`, `selector_name`, `selector_sku`;
- снизить нагрузку и добавить задержку в `page_delay_ms`.

### 5) В очереди много ошибок (`status=error`)

Порядок действий:

- отфильтровать ошибки в очереди;
- исправить селекторы/URL;
- выполнить `reset` (вернуть записи в `pending`);
- запустить парсинг повторно.

### 6) Сопоставление товаров не стартует или "зависло"

Если приходит ошибка, что сопоставление уже идет:

- проверьте `GET /api/matches/status?my_site_id=...`;
- остановите задачу через `POST /api/matches/stop`;
- при необходимости запустите продолжение через `POST /api/matches/retry-last`.

Для больших объемов:

- уменьшите `batchSize`;
- увеличьте `batchPauseMs`;
- используйте фильтрацию по `productSearch` или выбор конкретных товаров.

### 7) После синхронизации "моих товаров" часть позиций неактивна

Это нормальная логика:

- перед синхронизацией позиции помечаются `is_active=0`;
- найденные в источнике обновляются и становятся `is_active=1`;
- отсутствующие в источнике остаются неактивными.

### 8) Не работает синхронизация с МойСклад

Проверьте токен:

- `MS_TOKEN` в окружении или `msToken` в `config.js`;
- наличие прав доступа к API МойСклад.

Контрольные точки:

- запуск: `POST /api/ms/sync`;
- статус: `GET /api/ms/status`;
- результат: `GET /api/ms/export`.

### 9) После деплоя фронтенд показывает старую версию

Сделайте hard refresh браузера:

- macOS: `Cmd + Shift + R`

И проверьте, что сервер перезапущен (`pm2 restart parser-app`) и подтянул свежий commit.

### 10) Высокая нагрузка CPU у MySQL, долгие `SELECT`

Разделение по смыслу:

- **Очередь страниц** (`pages`, «Вернуть в очередь») — отдельные запросы к таблице `pages` и фоновый парсинг; не путать с товарами.
- **Статистика товаров** — `GET /api/my-products/stats` (в processlist часто видно начало `SELECT ... mp.site_id`). Тяжёлые варианты снимаются актуальной версией приложения (агрегация со связью к `ms_export` через индексируемое сравнение по `code` + короткий серверный кэш ответа). После обновления кода **обязательно перезапустите Node**, а не только MySQL.
- В phpMyAdmin для `SHOW FULL PROCESSLIST` может появиться предупреждение, что у результата нет уникального столбца — это ограничение UI, на сервер не влияет.

Диагностика: `SHOW FULL PROCESSLIST` (полный текст колонки `Info`), при необходимости `EXPLAIN` для конкретного запроса.
