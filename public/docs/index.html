Справка Datagon

Это веб-справка приложения Datagon: тексты в docs-docusaurus/docs/, после npm run docs:docusaurus:build попадают в public/docs/ и открываются по адресу /docs/ (обычно после входа в панель). Ниже — обзор продукта, как пользоваться справкой, пошаговые сценарии, глоссарий и развёрнутый FAQ.

Скриншот manual.png (главная справки). Без cookie автосъёмка не открывает живой /docs/ (цепочка session-bridge), поэтому в репозитории для этой картинки всегда берётся макет /doc-screenshots/manual-sample.html — он проще, чем та же справка у вас в браузере.

Скрины панели ниже обновляются по инструкции Скриншоты для справки: с DOCS_USER и DOCS_PASSWORD — с живых страниц вида /dashboard.html и т.д.; без входа — с макетов в каталоге /doc-screenshots/ (файлы …-sample.html); у длинных таблиц кадр — viewport. Только панель без макетов: npm run docs:capture-screenshots:real.

Дашборд. Кадр обновляется вместе с остальными PNG (съёмка): с входом — /dashboard.html, без — макет dashboard-sample.html; в кадре — полная страница этого экрана.

Для чего нужен Datagon​

Система помогает команде отвечать на вопросы:

  • какие цены у конкурентов по нашим SKU и артикулам;
  • что в наличии у нас и у конкурентов (по данным парсинга);
  • какие пары товаров уже сопоставлены автоматически и что нужно проверить вручную;
  • готовы ли данные к работе с МойСклад (ms_export).

Как пользоваться этой справкой​

  1. Боковое меню «Разделы» — дерево всех страниц; на мобильном открывается кнопкой «гамбургер» в шапке.
  2. Поиск в навбаре — локальный индекс по тексту страниц (нужна актуальная сборка, чтобы существовал search-index.json).
  3. Содержание справа (на широком экране) — якоря по заголовкам текущей страницы.
  4. Переключатель светлая/тёмная тема — в шапке (если не отключён темой ОС).
  5. Термины вроде my_products, очередь, матчинг — см. отдельную страницу Глоссарий.

Основной сценарий (развёрнуто)​

Шаг 1. Конкуренты​

Создайте проект для каждого конкурента: домен, селектор цены (обязателен), при возможности — название, SKU, признак «нет в наличии». Проверьте селекторы на одной реальной карточке в DevTools браузера. Подробно: Конкуренты.

Шаг 2. Очередь URL​

Добавьте ссылки на карточки товаров (вручную, пакетом или через обход sitemap — если доступно). Убедитесь, что URL относятся к правильному проекту. Подробно: Очередь.

Шаг 3. Парсинг и результаты​

Запустите парсинг для выбранных или отфильтрованных строк. Смотрите статус в очереди; при error — текст причины, правка селекторов, reset в pending и повтор. Итоговые строки — в Результаты.

Шаг 4. Мои сайты​

Подключите источник (вашу БД), проверьте маппинг полей, сохраните и выполните синхронизацию в my_products. Подробно: Мои сайты.

Шаг 5. Сопоставление​

Выберите свой сайт и конкурентов, задайте порог и размер пакетов, запустите матчинг. Разберите pending-кандидатов. При «зависании» — Логи и FAQ ниже. Подробно: Сопоставление.

Шаг 6. Мои товары​

Контролируйте активность, связь с МС, разрыв цен с конкурентами, точечные действия по строкам. Подробно: Мои товары.

Шаг 7. МойСклад (по необходимости)​

Запустите синхронизацию с API МС, дождитесь завершения, проверьте ms_export. Подробно: МойСклад.

Краткая цепочка экранов: Конкуренты → Очередь → Результаты → Мои сайты → Сопоставление → Мои товары → МойСклад.

Сценарии «первый день»​

  1. Только конкуренты (без своего каталога): Конкуренты → Очередь → Результаты. Настройте задержки в Настройках, следите за Логами, если включён массовый парсинг.
  2. Свой каталог из БД: Мои сайты (подключение и синк) → Мои товары. Затем при необходимости МойСклад для сверки с учётной системой.
  3. Цены «мы vs конкурент»: п.1 + актуальные my_products (п.2) → Сопоставление (ручное подтверждение пар) → снова Мои товары для колонок разрыва.
  4. Продакшен после установки: Деплой, проверка curl из FAQ, смена пароля admin, токен МС в config.js / MS_TOKEN.

Технологии и репозиторий​

  • Сервер: Node.js, Express, MySQL, Axios, Cheerio, bcrypt.
  • Панель: статика в public/ (ArchitectUI Bootstrap).
  • Справка: Docusaurus 3; сборка: npm run docs:docusaurus:build.

Быстрый старт (локально)​

  1. git clone репозитория, cd в каталог проекта.
  2. npm install.
  3. Скопируйте/настройте config.js: MySQL (host, user, password, database), port, при необходимости токен МойСклад.
  4. npm start, откройте http://localhost:<port>/login.html или дашборд.
  5. Документацию собрать: npm run docs:docusaurus:build (опционально).

После правок серверного кода перезапустите процесс Node — см. раздел «Важные замечания» ниже.

FAQ и типовые проблемы​

1) Не удаётся войти​

Проверьте:

  • URL панели: …/dashboard.html или корень / с редиректом;
  • запущен ли npm start (или pm2 на сервере);
  • логин и пароль; смена пароля через UI или API.

Проверка API:

curl -i -X POST "http://localhost:3000/api/auth/login" \
  -H "Content-Type: application/json" \
  --data '{"username":"YOUR_USERNAME","password":"YOUR_PASSWORD"}'

2) Приложение не стартует: ошибка БД​

В config.js: db.host, db.user, db.password, db.database; доступность MySQL и права; существование базы. В консоли ищите [DB] Fatal Error. Исправьте параметры и перезапустите Node.

3) Источник не добавляется (DB Connect)​

Обычно: внешняя БД недоступна с сервера Datagon, неверный хост/порт, имя базы, таблица товаров, поля SKU/цены/остатка. Для Webasyst — отдельная таблица SKU и поля wa_*. См. Мои сайты.

4) Парсинг не даёт данных​

Частые причины: неверные селекторы; страница не карточка товара; цена только в JS (в сыром HTML нет); лимиты/бан со стороны сайта.

Действия: проверить одну страницу через API POST /api/pages/page/:id; заполнить selector_price, selector_name, selector_sku в проекте; увеличить page_delay_ms в Настройках.

5) В очереди много error​

Отфильтровать ошибки, прочитать причину, исправить URL/селекторы, выполнить reset в pending, снова запустить парсинг. См. Очередь.

6) Сопоставление не стартует или «зависло»​

Если ответ API говорит, что задача уже идёт:

  • GET /api/matches/status?my_site_id=…
  • POST /api/matches/stop с телом { "mySiteId": … }
  • при необходимости POST /api/matches/retry-last

На больших объёмах уменьшите batchSize, увеличьте batchPauseMs, сузьте выборку (productSearch, список productIds). См. Сопоставление.

7) После синка часть «моих товаров» неактивна​

Это нормально: перед синком позиции помечаются is_active=0; найденные в источнике становятся 1; отсутствующие в источнике остаются неактивными.

8) Не работает МойСклад​

Токен: переменная MS_TOKEN или msToken в config.js, права API МС. Контроль: POST /api/ms/sync, GET /api/ms/status, GET /api/ms/export. См. МойСклад.

9) После деплоя старый фронт​

Жёсткое обновление страницы (Cmd+Shift+R на macOS), git pull на сервере, pm2 restart (или ваш процесс), убедиться что выложен актуальный public/. Чеклист: Деплой.

10) Высокая нагрузка MySQL​

Не путать очередь страниц (pages) и тяжёлые списки товаров / GET /api/my-products/stats. После обновления кода с оптимизациями — перезапуск Node, не только MySQL. Диагностика: SHOW FULL PROCESSLIST, при необходимости EXPLAIN.

Важные замечания​

  • Таблицы БД создаются и дорабатываются при старте server.js.
  • Совместимость: POST /api/login (legacy) и POST /api/auth/login.
  • После изменений server.js, routes/, lib/ нужен перезапуск Node — иначе API в браузере останется старым.
